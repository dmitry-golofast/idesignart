# idesignart — Референсы, стратегия и структура сайта

> **Стек:** Nuxt 4 · Payload 3 (headless) · Tailwind CSS v4 · TypeScript
> **Дизайн-система:** Nuxt UI v4 (foundation) + кастомные премиум-блоки (shadcn-подход)

> Рабочий документ: подбор сайтов-референсов + критерии оценки + структура
> страниц + SEO/лидогенерация для студии дизайна и 3D-визуализации.
> Цель — сайт, который приводит клиентов на дизайн/визуализацию квартиры,
> интерьера, офиса, пространства.

## 3. Структура сайта (Information Architecture)

### 3.1. Карта страниц

```
idesignart.com
│
├── / (Главная)
│
├── /services/                    ← концентратор, линкует на посадочные
│   ├── /services/interior-design/          — Дизайн интерьера
│   ├── /services/apartment-design/         — Дизайн квартиры
│   ├── /services/office-design/            — Дизайн офиса / рабочего пространства
│   ├── /services/3d-visualization/         — 3D-визуализация интерьера ⭐
│   ├── /services/space-planning/           — Планировочные решения
│   └── /services/author-supervision/       — Авторский надзор
│
├── /portfolio/                   — список проектов
│   └── /portfolio/[project-slug]/         — страница кейса (с текстом для SEO)
│
├── /pricing/                     — тарифы / калькулятор / диапазоны
│
├── /process/                     — «как мы работаем» (3–5 шагов)
│
├── /about/                       — о студии, команда, ценности
│
├── /blog/                        — статьи, гайды (SEO-движок)
│   └── /blog/[article-slug]/
│
├── /contacts/                    — форма + мессенджеры + онлайн-запись
│
├── /case-studies/                — (опционально, отдельно от портфолио)
│
└── юридические: /privacy/, /terms/
```

> **Почему так много посадочных под услуги:** каждый тип запроса
> («дизайн кухни», «3D-визуализация квартиры», «дизайн офиса») — это
> отдельный интент. Одна общая страница «услуги» проиграет в SEO
> специализированным конкурентам.

### 3.3. Структура страницы услуги (landing page)

Каждая страница услуги (`/services/...`) строится по шаблону:

1. **Hero** — H1 с ключевым словом + подзаголовок с выгодой + CTA.
2. **Боли клиента** — 3 пункта «что вас беспокоит» (снимает возражения).
3. **Решение** — что именно делаете, 3–5 пунктов с иконками.
4. **Процесс** — те же 4 шага, адаптированные под услугу.
5. **Портфолио по этой услуге** — только релевантные проекты (фильтр по тегу).
6. **Отзывы по этой услуге**.
7. **FAQ** — 5–8 вопросов (микроразметка `FAQPage` = расширенный сниппет в выдаче).
8. **Цена/диапазон**.
9. **Финальный CTA + форма**.

> Такой landing решает **одну задачу** — конвертировать трафик по
> конкретному интенту. Не пытайтесь продать всё сразу на одной странице.

### 3.4. Структура страницы кейса (портфолио)

Это **самая важная SEO-страница** — она приносит long-tail трафик.

1. Hero с обложкой проекта.
2. Метаданные: площадь, стиль, срок, бюджет (диапазон), локация.
3. **Задача клиента** (текст — это SEO-контент!).
4. **Решение** — что сделали, почему так.
5. Галерея «до/после» + 3D-визуализации.
6. Использованные материалы/бренды (можно партнёрские ссылки).
7. Отзыв клиента.
8. «Похожие проекты» (перелинковка).
9. CTA «Хотите так же?».

## 4. SEO-стратегия

### 4.1. Семантическое ядро (структура по кластерам)

| Кластер | Пример запросов | Посадочная |
|---------|-----------------|------------|
| **Дизайн квартиры** | «дизайн квартиры», «дизайн интерьера квартиры», «дизайн 2-комнатной квартиры», «дизайн кухни-гостиной» | `/services/apartment-design/` |
| **Дизайн офиса** | «дизайн офиса», «оформление рабочего пространства», «дизайн переговорной» | `/services/office-design/` |
| **3D-визуализация** | «3D визуализация интерьера», «заказать 3D проект квартиры», «визуализация дизайна интерьера» | `/services/3d-visualization/` |
| **Региональные** | «дизайн интерьера [город]», «дизайнер интерьера [район]» | страницы городов или блоки на услугах |
| **Информационные** | «как выбрать дизайнера», «сколько стоит дизайн интерьера», «стили интерьера» | блог |

### 4.2. Технический SEO-чеклист

- [ ] Уникальные `<title>` и `<meta description>` на каждой странице (с ключевым словом + CTA).
- [ ] Чистая иерархия заголовков (один H1, далее H2/H3).
- [ ] ЧПУ (человекопонятные URL): `/services/3d-visualization/`, а не `/?p=123`.
- [ ] `sitemap.xml` + `robots.txt`.
- [ ] Микроразметка: `LocalBusiness`, `Service`, `FAQPage`, `BreadcrumbList`, `Review`, `ImageObject`.
- [ ] Open Graph + Twitter Card для шаринга.
- [ ] Alt-тексты ко всем изображениям (особенно рендерам!).
- [ ] `hreflang`, если мультиязычность.
- [ ] Canonical на дублирующихся страницах.
- [ ] HTTPS, мобильная адаптивность, скорость (Core Web Vitals).

### 4.3. Контентная стратегия (блог)

Форматы, которые приносят трафик и лиды:

1. **Гайды «как выбрать»** — «Как выбрать дизайнера интерьера: 10 вопросов».
2. **Калькуляторы и чек-листы** — «Чек-лист ремонта: 50 пунктов».
3. **Разборы стилей** — «Скандинавский стиль: 7 признаков + 20 примеров».
4. **Кейсы с цифрами** — «Дизайн кухни-гостиной 45 м²: бюджет, сроки, результат».
5. **Сравнения** — «3D-визуализация vs фотосъёмка: что выбрать».

> Частота: 2–4 статьи в месяц. Каждая статья → оптимизирована под
> 1 ключевое слово, 1500–3000 слов, с уникальными изображениями.

## 5. Технические требования (продакшен)

### 5.1. Производительность

| Метрика | Цель | Как |
|---------|------|-----|
| Lighthouse Performance (mobile) | ≥ 90 | Оптимизация изображений, минимальный JS |
| LCP | < 2.5s | Lazy-load ниже первого экрана, приоритетный hero |
| CLS | < 0.1 | Заданные размеры изображений (`width`/`height` или `aspect-ratio`) |
| INP | < 200ms | Минимум блокирующего JS, code-splitting |

**Форматы изображений:**
- AVIF / WebP (с fallback JPEG).
- Responsive `srcset` под разные экраны.
- Видео через streaming (HLS), **не** тяжёлыми `.mp4`.
- Lazy-load всего вне первого экрана.

### 5.2. Accessibility (WCAG 2.1 AA)

- [ ] Контраст текста ≥ 4.5:1 (обычный текст), ≥ 3:1 (крупный).
- [ ] Все интерактивные элементы доступны с клавиатуры (фокус-стили видны).
- [ ] Alt-тексты к изображениям.
- [ ] Семантический HTML (`<nav>`, `<main>`, `<button>`, а не `<div>`).
- [ ] `aria-label` на иконочных кнопках.
- [ ] Масштабируемость до 200% без поломки вёрстки.

### 5.3. Стек (Nuxt + Payload + Tailwind + TypeScript)

| Слой | Технология | Назначение |
|------|------------|------------|
| **Фреймворк** | Nuxt 4 | SSR/SSG/ISR для SEO, file-based routing, auto-imports, `<NuxtImg>` для оптимизации изображений |
| **CMS** | Payload 3 (headless) | Управление портфолио, блогом, услугами. Отдаёт REST/GraphQL API |
| **Стили** | Tailwind CSS v4 | Utility-first, design tokens через CSS-переменные (`@theme`) |
| **Язык** | TypeScript (strict) | Типобезопасность на фронте + типы из Payload-схем |
| **UI-библиотека** | Nuxt UI v4 (на Reka UI) | 125+ accessible-компонентов, полная Tailwind v4 интеграция (см. п. 5.4) |
| **Дизайн-система** | Собственная, на базе Nuxt UI Pro tokens | Премиальная типографика, палитра, отступы (см. п. 5.5) |
| **SEO** | `@nuxtjs/sitemap` + `@nuxtjs/robots` + `nuxt-schema-org` | Sitemap, robots, JSON-LD микроразметка |
| **Изображения** | `@nuxt/image` + Payload Media | AVIF/WebP, responsive srcset, lazy-load |
| **Аналитика** | GA4 + Яндекс.Метрика + Plausible | GA4/Метрика — рекламная аналитика, Plausible — скорость и privacy |
| **Формы** | Payload Form Builder + `vee-validate` + `zod` | Лид-формы с валидацией и отправкой в Payload |

#### Архитектура Payload + Nuxt

```
┌──────────────┐         REST/GraphQL          ┌──────────────┐
│  Nuxt 4      │  useFetch() / useAsyncData()  │  Payload 3   │
│  (frontend)  │ ◄───────────────────────────► │  (headless)  │
│              │                                │              │
│  SSR/SSG     │                                │  Admin panel │
│  Tailwind v4 │                                │  REST API    │
│  Nuxt UI v4  │                                │  GraphQL API │
└──────────────┘                                │  Local API   │
                                                └──────┬───────┘
                                                       │
                                                ┌──────▼───────┐
                                                │  PostgreSQL  │
                                                └──────────────┘
```

**Как связывать:**
- Payload запускается как отдельный Node-сервер (или в монорепо через `apps/payload`, `apps/web`).
- Nuxt забирает данные через `useFetch` (REST) или `useAsyncData` + Apollo/`@nuxtjs/apollo` (GraphQL).
- Типы генерируются из Payload-схем → `payload generate:types` → импорт в Nuxt (`types/payload.ts`).
- Превью драфтов: Payload `preview` + Nuxt route `/api/preview` с cookie-проверкой.
- Webhooks: при публикации контента в Payload → ревалидация ISR-страниц в Nuxt (`revalidatePath`-аналог через `nitro` хуки).

#### Почему не Next.js

Next.js — нативный стек Payload (admin-панель встроена). Но выбор Nuxt обоснован, если:
- Команда сильнее во Vue-экосистеме.
- Нужен `@nuxt/image` и авто-imports для скорости разработки.
- Хочется SSR-гибкость Nuxt (Nitro-сервер, deploy на любой edge).

Платой — ручная интеграция Payload (headless), но это разовая работа.

### 5.4. Дизайн-система: сравнение и выбор

> **Контекст:** нужен премиальный, сильно кастомизированный визуал
> (как Studio Ashby / Norm Architects), а не стандартный SaaS-look.
> При этом — accessibility и скорость разработки.

#### Кандидаты

| Критерий | **Nuxt UI v4** | **shadcn-nuxt** | **DaisyUI** |
|----------|----------------|-----------------|-------------|
| Архитектура | Runtime-зависимость | Local (код в репо) | Tailwind-плагин |
| Tailwind v4 | ✅ Нативно | ✅ Через CSS-переменные | ✅ |
| Базис accessibility | Reka UI (Radix Vue) | Reka UI (Radix Vue) | ⚠️ Базовый |
| Кол-во компонентов | 125+ (вкл. бывший Pro) | ~50 (добавляешь сам) | ~30 классов |
| Скорость старта | ⭐⭐⭐⭐⭐ Из коробки | ⭐⭐⭐ Ручная сборка | ⭐⭐⭐⭐⭐ |
| Кастомизация | ⭐⭐⭐ Через tokens/API | ⭐⭐⭐⭐⭐ Полный контроль над кодом | ⭐⭐ themes |
| Премиальный look | ⭐⭐⭐ Нужен ребрендинг | ⭐⭐⭐⭐ Полный контроль | ⭐⭐ Шаблонный |
| Обновления | npm update | Ручной merge ваших компонентов | npm update |
| Владение кодом | В node_modules | В вашем репо ✅ | В node_modules |

#### Рекомендация: гибридный подход ⭐

**Nuxt UI v4 как фундамент + shadcn-подход для ключевых премиум-компонентов.**

Логика:
1. **Базовые интерактивные компоненты** (Button, Modal, Dropdown, Tabs, Accordion, Form inputs) → **Nuxt UI v4**. Это даёт гарантированную accessibility (Reka UI), Tailwind v4 и скорость. Переопределяем через design tokens (палитра, скругления, тени) — не трогая код компонентов.
2. **Презентационные премиум-блоки** (Hero, ProjectCard, ProcessSteps, PricingCalculator, TestimonialCard, PortfolioMasonry) → **свои компоненты в репо** (как shadcn-философия). Полный контроль над вёрсткой под референсы Studio Ashby / Maverick Frame.
3. **Дизайн-токены** → единый источник правды в `@theme` (Tailwind v4 CSS-переменные). Nuxt UI и кастомные компоненты берут значения оттуда → консистентность.

> Это даёт: скорость Nuxt UI на рутине + полную свободу на «продающих»
> блоках, где важен уникальный премиальный вид.

### 5.5. Дизайн-токены (foundation)

Единый файл `assets/css/tokens.css` с Tailwind v4 `@theme`:

#### Палитра

```css
@theme {
  /* Нейтральная база — тёплый графит (как Norm Architects) */
  --color-canvas:    #FAF8F5;   /* фон, тёплый белый */
  --color-surface:   #FFFFFF;   /* карточки */
  --color-ink:       #1A1816;   /* основной текст, графит */
  --color-ink-muted: #6B6660;   /* вторичный текст */

  /* Акцент — терракота (как Studio Ashby) */
  --color-accent:        #B5533C;  /* CTA, активные элементы */
  --color-accent-soft:   #E8D5CC;  /* фоны-подложки */
  --color-accent-deep:   #7A2E1F;  /* hover */

  /* Состояния */
  --color-success: #4A7C59;
  --color-error:   #B5533C;
  --color-focus:   #1A1816;  /* фокус-кольцо */
}
```

#### Типографика

```css
@theme {
  /* Заголовки — оверсайз-сериф (премиальность) */
  --font-display: "Fraunces", "Cormorant", Georgia, serif;

  /* Текст — чистый гротеск (читаемость) */
  --font-sans: "Inter", "Manrope", system-ui, sans-serif;

  /* Размерная шкала (clamp — адаптив без брейкпоинтов) */
  --text-display: clamp(2.75rem, 6vw, 6rem);    /* H1 hero */
  --text-h1:      clamp(2.25rem, 4vw, 3.5rem);
  --text-h2:      clamp(1.75rem, 3vw, 2.5rem);
  --text-h3:      clamp(1.25rem, 2vw, 1.5rem);
  --text-body:    1rem;
  --text-small:   0.875rem;
  --text-micro:   0.75rem;       /* лейблы, метаданные */

  --tracking-tight:  -0.03em;  /* для display */
  --tracking-normal: 0;
  --tracking-wide:   0.08em;   /* лейблы, капс */
}
```

#### Отступы, скругления, тени

```css
@theme {
  /* Скругления — мягкие, премиум */
  --radius-sm:   4px;
  --radius-md:   8px;
  --radius-lg:   16px;
  --radius-pill: 9999px;

  /* Тени — мягкие, тёплые (не дефолтные серые) */
  --shadow-card:  0 2px 8px rgba(26,24,22,0.06), 0 8px 24px rgba(26,24,22,0.04);
  --shadow-hover: 0 8px 24px rgba(26,24,22,0.12), 0 16px 48px rgba(26,24,22,0.08);

  /* Контейнеры и сетка */
  --container-max: 1440px;
  --container-pad: clamp(1rem, 4vw, 3rem);

  /* Переходы — медленные, дорогие (премиум feel) */
  --ease-premium: cubic-bezier(0.22, 1, 0.36, 1);
  --duration-base: 0.4s;
  --duration-slow: 0.7s;
}
```

> Все значения в одном месте → и Nuxt UI, и кастомные компоненты
> ссылаются на них. Сменить акцентный цвет = правка одной строки.
